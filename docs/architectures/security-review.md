# [アーキテクチャ](./index.md) - セキュリティレビュー指針

## 目的

このドキュメントは、変更内容に応じて確認すべきセキュリティ観点を整理し、レビュー時に[セキュリティチェックリスト](./security-checklist.md)へ迷わず辿れるようにすることを目的とする。

## 基本方針

- すべての変更で、入力、認証、ログ、設定、依存関係への影響有無を確認する
- セキュリティ上の判断に迷う場合は実装を止め、懸念点と不足情報を共有する
- リファクタリングやドキュメント更新でも、既存のセキュリティ前提を壊していないかを確認する

## 確認の進め方

1. 変更対象がどの領域に影響するかを整理する
2. 本ドキュメントの該当節で重点確認項目を確認する
3. 詳細な抜け漏れ確認は[セキュリティチェックリスト](./security-checklist.md)の該当節で行う
4. 必要に応じてテストを追加し、回帰がないことを検証する

## 重点確認項目

### インプット／インジェクション対策

- SQL、NoSQL、コマンド、テンプレート出力で文字列連結による危険な組み立てをしていないか確認する
- ユーザー入力や外部入力は、型、形式、長さを検証し、受け入れ条件を明確にする
- JSON や HTML への出力ではエスケープ処理を確認し、危険な値をそのままログへ出さない

関連チェックリスト: [入力検証・サニタイゼーション](./security-checklist.md#-入力検証サニタイゼーション)

### 認証・認可

- `features/auth` に依存する改修では、登録、認証、トークン更新、サインアウトの流れを順番に確認する
- 認可判定をクライアントだけに依存させず、サーバー側でもロールやスコープを再検証する
- セッション、トークン、クッキーを扱う場合は、有効期限、失効、再利用防止の扱いを確認する

関連チェックリスト: [認証・認可](./security-checklist.md#-認証認可)

### ログ・監査

- ログやエラーメッセージに認証情報、個人情報、シークレットが出力されないことを確認する
- セキュリティイベントの追跡に必要な監査情報を失っていないか確認する
- マスキングや監査ログの扱いを変更した場合は、運用上の影響を明確にする

関連チェックリスト: [ログ・監視](./security-checklist.md#-ログ監視)

### 依存関係・設定変更

- 新規依存関係の追加時は、脆弱性対応方針と更新可能性を確認する
- CSP、HTTP ヘッダー、CORS、ACL、環境変数などの設定変更が既存防御を弱めていないか確認する
- インフラや実行環境に関わる変更では、必要な最小権限が維持されているか確認する

関連チェックリスト:

- [依存関係管理](./security-checklist.md#-依存関係管理)
- [設定管理](./security-checklist.md#-設定管理)
- [ネットワークセキュリティ](./security-checklist.md#-ネットワークセキュリティ)

## レビュー時の判断基準

- 不明な入力経路が増えていないか
- 認証前提や権限制御の責務境界が崩れていないか
- 失敗時のログやレスポンスに過剰な情報が含まれていないか
- 既存テストでは担保できないリスクが増えた場合、追加テストが必要か
