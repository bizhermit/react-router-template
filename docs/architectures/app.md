# [アーキテクチャ](./index.md) - アプリケーション構成

## 目的

このドキュメントは、本アプリケーションの構成要素と責務境界を明確にし、実装・レビュー時の判断基準を提供することを目的とする。

## 関連ドキュメント

- 実際の配置先とルールは[ディレクトリ構成](./directory.md)を参照
- 配置判断の起点は[追加時の判断フロー](./directory.md#追加時の判断フロー)

## 全体像

本アプリケーションは、ReactRouterを中心としたSSR/CSRハイブリッド構成である。  
画面のルーティングとUIは`src/app`、ユースケースに依存する機能は`src/features`、汎用基盤は`src/lib`に分離し、依存方向を一貫させる。

## レイヤー構成

各レイヤーの具体的な配置例は[ディレクトリ詳細](./directory.md#ディレクトリ詳細)を参照する。
本ドキュメントで定義する4層はアプリケーションの責務構成を示すものであり、[コーディング規約](./code.md)の「役割・依存方向・実行環境」の3軸は、その各層の実装配置と依存関係を判断するための補助基準である。

### 1. プレゼンテーション層（`src/app`）

- ルーティング定義とページコンポーネントを管理する
- URL、loader/action、画面状態といったReactRouter依存の責務を持つ
- ユースケース実装は直接持たず、`src/features`または`src/lib`の公開インターフェースを呼び出す

### 2. 機能層（`src/features`）

- 認証など、アプリ固有のユースケースを機能単位で管理する
- 機能ごとに`client`/`server`/`shared`を分離し、実行環境ごとの責務を明確にする
- 機能内の依存は同一機能グループ内と`src/lib`に限定する

### 3. 基盤層（`src/lib`）

- アプリ非依存の共通処理（UI部品、共通ユーティリティ、i18n、HTTP、スキーマ等）を提供する
- 可能な限り副作用を局所化し、再利用しやすい構成を維持する
- `src/features`への逆依存を禁止する

### 4. データアクセス層

- DBスキーマ定義は`src/features/database/server/schema.ts`および`drizzle/`で管理する
- サーバーサイド実装からのみDBへアクセスし、クライアントからの直接アクセスは禁止する
- データアクセス層は論理上の責務区分であり、独立したトップレベルディレクトリを新設することを意味しない。実装は`src/features/*/server`、`src/lib/server`、`drizzle/`に配置する

## リクエスト処理フロー

1. ブラウザからHTTPリクエストを受信する
2. `src/app/routes.ts`に基づき対象ルートを解決する
3. 対応するroute moduleの`loader`/`action`/Componentを実行する
4. 必要に応じて`src/features/*/server`のユースケースを呼び出す
5. ユースケース内で`src/lib/server`やDBアクセスを実行する
6. 返却データをroute moduleで整形し、SSR/CSR描画へ渡す

## 依存ルール

- 許可する依存方向は`app -> features -> lib`および`app -> lib`とする
- `lib -> features`の依存は禁止する
- `server`専用モジュールは`client`からimportしない
- `client`専用モジュールは`server`処理の中核ロジックに組み込まない
- `shared`は環境非依存の処理のみ配置する

依存ルールに沿った配置判断は[配置ルール](./directory.md#配置ルール)で確認する。

## 機能境界

- `src/app`: 画面遷移、URL解釈、表示制御
- `src/features/auth`: 認証・認可関連のユースケース
- `src/features/api`: アプリ固有のAPI連携
- `src/lib/components`: 機能に依存しないUI部品
- `src/lib/shared`: 環境共通のユーティリティ

## 横断関心事

### 認証・認可

- 認証フローは`src/features/auth`で管理する
- 認可判定はクライアントだけで完結させず、サーバー側で再検証する

### エラーハンドリング

- ルート境界のエラー処理は`src/app/routes/fallback.tsx`を中心に統一する
- 外部向けエラーメッセージには機密情報を含めない

### ログ・監査

- ログ出力時は個人情報・認証情報をマスクする
- 開発時と本番時でログレベルを切り替え可能な設計を維持する

### i18n

- 翻訳リソースは`public/locales`で管理する
- 画面文字列はハードコードを避け、i18n経由で解決する

## 外部依存と接続点

- BetterAuth: 認証基盤
- PostgreSQL + Drizzle: 永続化
- OpenAPI関連ファイル（`docs/api`）: API仕様の参照元

外部依存への接続は可能な限り`server`層に集約し、UI・ルート層から直接扱わない。

## 既知の制約

- `src/app`配下のroute moduleはReactRouterへの依存が高く、他アプリへの移植性は低い
- 機能追加時に`lib`へ責務を寄せすぎると、機能境界が曖昧になりやすい

## 今後の拡張方針

- 機能追加時は`src/features/<feature-name>`を起点に責務を閉じる
- 外部サービス連携は`server`側に集約し、境界に型安全なI/Oスキーマを置く
- 依存ルール違反を防ぐため、lintまたはレビュー観点の明文化を継続する

実装時の配置に迷った場合は、都度[ディレクトリ構成](./directory.md)の手順に戻って確認する。
